# 02. 权限配置：让本地 Codex 直接写飞书文档 / 多维表格

这一篇讲权限，不讲 bridge 本身。

前一篇解决的是“怎么接进飞书”。  
这一篇解决的是“接进去以后，怎么让它真的能写我的飞书文档，后面再继续碰多维表格”。

## 一、先把问题分清

这里有两个完全不同的层面：

### 1.1 能在飞书里聊天

这说明：

- bridge 通了
- 飞书消息能进本地 agent
- 本地 agent 能回飞书

### 1.2 能直接写你的飞书文档

这说明：

- 当前身份正确
- 当前 token 有效
- 对应 scope 配齐了
- 对目标文档确实有写权限

这两件事不能混为一谈。

## 二、为什么飞书链接不等于授权

这个地方单独说一句就够了：

你把飞书文档链接发给 Codex，不等于它就继承了你的登录态。

链接只能说明文档存在，不能说明：

- 它能读
- 它能写
- 它能代表你的账号去操作这个文档

所以后面还是得走正式授权。

## 三、为什么这里要用 `lark-cli`

因为真正负责身份检查、scope 检查、文档读写动作的，就是 `lark-cli`。

它在这条链路里不是辅助工具，而是主工具。

后面这些动作都会用它做：

1. 看当前是不是用户身份
2. 看 token 是否有效
3. 看 scope 是否够
4. 读飞书文档
5. 写飞书文档

## 四、先看飞书应用权限

在飞书开放平台授权时，能直接看到它申请了哪些能力。

下面这张图就很直观：

![飞书开放平台授权页](../assets/04-feishu-auth.jpg)

这一步能看到的重点包括：

- 新版文档
- 云空间文件元数据
- 多维表格、电子表格、幻灯片的常用权限

如果这些范围一开始就没配，后面再写命令也没用。

## 五、再看飞书侧智能体页面

这张图更适合放在“应用本身已经建好”的位置：

![飞书智能体页面](../assets/03-xiaohai-profile.jpg)

它能说明：

- 这是哪个智能体
- 开发者是谁
- 当前权限入口是怎么挂上的

这张图不等于写权限已经打通，但很适合用来说明“这个应用已经建起来了”。

## 六、手机扫码授权这一步怎么做

这里说的是 **给 `lark-cli` 做用户身份授权**，不是前一篇里第一次绑定 Agent 的扫码。

这一步是为了让本地 Codex 后面能按用户身份去读写飞书文档。

### 6.1 先在终端里拿授权链接和 device code

我当时终端里先跑的是这条：

```bash
lark-cli auth login --domain docs --domain drive --domain wiki --no-wait --json
```

这一步会返回两样关键东西：

1. `verification_url`
2. `device_code`

### 6.2 把授权链接转成二维码给手机扫

拿到 `verification_url` 以后，可以继续用 `lark-cli auth qrcode` 生成二维码，让手机直接扫。

我当时那次就是终端先拿授权链接，再把它转成二维码发出来给手机扫码。

如果你在 Windows 下直接走 `.cmd` 包装层，链接里有 `&` 时很容易被拆坏。  
更稳的做法是直接调用真正的 `lark-cli.exe`，不要让 `.cmd` 去解析这个 URL。

### 6.3 手机端看到的就是这类授权页

这张图就是你当时手机上看到的授权页：

![飞书开放平台授权页](../assets/04-feishu-auth.jpg)

这里要做的动作很简单：

1. 确认应用没错
2. 看一下权限范围
3. 点“开通并授权”

### 6.4 手机上授权完成后，回终端收尾

手机点完授权以后，还不算彻底结束。

终端这边还要再执行一次：

```bash
lark-cli auth login --device-code <device_code>
```

这样这次授权才会真正写回本地 CLI 配置。

## 七、授权检查流程

这里建议直接按顺序做。

### 7.1 检查当前身份

```bash
lark-cli whoami
lark-cli auth status --json --verify
```

主要看三样：

1. `identity` 是不是 `user`
2. token 是否有效
3. scope 是否够

## 八、先读，再写

### 8.1 先读文档

```bash
lark-cli docs +fetch --doc "<你的飞书文档链接>" --as user
```

这一步确认的是：

- 当前身份对这个文档能不能读
- 当前目标文档有没有搞错

### 8.2 再做写入验证

推荐顺序：

1. 先 dry-run
2. 再追加一小段测试内容
3. 再读回去确认
4. 最后清掉测试段落

这样做的好处是不会一上来就把正式文档改乱。

## 九、为什么我这里更建议用户身份

如果你的目标是：

- 写我自己的飞书文档
- 读我自己的 wiki
- 后面继续接多维表格

那通常还是用户身份更顺。

因为 bot 身份更容易卡在“应用能运行，但不是你的账号在操作”这一层。

## 十、多维表格为什么可以接在这后面

这一步不用另起一套思路。

文档和多维表格虽然不是同一个资源，但从接入链路上看，可以共用这些基础：

1. 同一套飞书应用
2. 同一套授权状态
3. 同一套本地 Codex
4. 同一套 `lark-cli`

文档先打通以后，后面补多维表格时，主要补的是：

- Base / 多维表格目标
- 字段结构
- 记录写入逻辑

不是把整条链再重建一遍。

## 十一、建议的实践顺序

如果你是第一次配，建议直接按这个顺序走：

### 11.1 先把飞书 Agent 接通

也就是先完成前一篇里的 bridge 部分。

### 11.2 再确认 `lark-cli` 当前是用户身份

不要先入为主觉得已经是。

### 11.3 先对一个现有文档做 fetch

能读到，才说明这条链有意义。

### 11.4 再做一次小范围写入验证

不要一开始就全量覆盖正式文档。

### 11.5 文档通了，再继续扩到多维表格

这样比较稳。

## 十二、这一篇里最容易卡住的 4 个点

### 12.1 bridge 通了，但身份不对

这时候飞书里也能聊天，但文档写不了。

### 12.2 能读，不能写

这通常说明 scope 或写权限还没配齐。

### 12.3 scope 少了文档 / wiki / Base 相关权限

这类问题不补权限，后面命令不会自己变通。

### 12.4 想一步把文档和多维表格全做完

更稳的方式还是先把文档打通，再继续做表格。
