# 02. 让本地 Codex 直接写飞书文档 / 多维表格

这一篇讲的是第二层问题：

**飞书里能和 Codex 对话，不代表当前这个本地 Codex 已经有权限替你写飞书。**

这个差别如果不先分清，后面会一直拧巴。

## 先把误区掐掉

### 误区 1：我把链接发给 Codex，不就等于授权了吗

不是。

飞书链接只是链接，不是你的账号，不是你的 token，也不是你的写权限。

你把一个 wiki 或 doc 链接发出来，只能说明“这个页面存在”，不能说明“当前这个 agent 已经能代表你去改它”。

### 误区 2：既然都在飞书里聊天了，它顺手就能写飞书吧

也不是。

聊天入口和写文档权限，压根不是一回事。

前者解决的是“消息怎么进来”，后者解决的是“动作有没有资格执行”。

## 为什么最后还是得靠 `lark-cli`

因为这条路最实。

你可以把 `lark-cli` 理解成：  
这套链路里真正负责“验证身份、检查 scope、执行读写动作”的工具层。

它不是点缀，也不是可有可无。

如果没有它，你很容易停在一种很尴尬的状态：

- 看起来好像连上了
- 也能在飞书里说话
- 但真让它改文档时，又总差半步

## 为什么我更建议用用户身份

这里只说结论：

如果你的目标是“让 Codex 能改我自己的飞书文档，后面还能碰多维表格”，那更稳的方向一般是用户身份，而不是只靠 bot 身份。

原因很简单。

bot 身份更像：

- 它自己的应用
- 它自己的权限范围
- 它自己的资源边界

而你要的通常是：

- 写你的文档
- 读你的 wiki
- 后面接你的表

这种场景下，用户身份通常更顺。

## 这条权限链到底要做什么

别把它想复杂，核心就是三步：

### 第一步：确认当前身份是不是对的

先查状态，不要上来就写。

典型命令就是：

```bash
lark-cli whoami
lark-cli auth status --json --verify
```

这里最该看的不是花里胡哨的输出，而是：

- 现在到底是不是 `user`
- token 到底是不是有效
- scope 到底够不够

如果这一步都没过，后面继续写命令也只是热闹。

### 第二步：先读文档

```bash
lark-cli docs +fetch --doc "<你的飞书文档链接>" --as user
```

这一步的意义不是“读给你看”，而是验证两件事：

1. 这个链接当前真的能被这套身份读到
2. 你后面准备写的目标，没有搞错

### 第三步：再写

写的时候也别一上来就全量覆盖。

更稳的做法是：

1. 先 dry-run
2. 再追加一小段测试内容
3. 再读回去确认
4. 最后把测试内容清掉

这样才算真的通。

不是“命令没报错”就叫通了。

## 为什么这一步和多维表格其实是同一路

这一点我觉得很值得单独说。

很多人一听“飞书文档”和“多维表格”，会默认这是两套完全不同的事情。

但从这类自动化接入的角度看，它们没有你想得那么裂。

因为真正要复用的东西，其实是这些：

- 同一套飞书应用
- 同一套授权状态
- 同一套本地 Codex
- 同一套“本地执行，飞书承接结果”的工作方式

换句话说，文档先打通，不是在做一次性工作，而是在把整条权限链铺平。

后面接 Base / 多维表格时，你补的是：

- 目标表
- 字段结构
- 记录写入逻辑

不是从头再造世界。

## 这一类问题最容易卡住的地方

这一段我不写成“踩坑复盘”，就直接写成判断清单。

### 1. 看起来连上了，其实身份不对

这时候最常见的情况就是：

- bridge 是通的
- 飞书里也能聊天
- 但 `whoami` 一看，不是你想要的身份

### 2. 能读，但不能写

这也很常见。

读成功只能说明“当前资源对这套身份可见”，不能自动推出“写权限也已经配齐”。

### 3. scope 不够

如果缺的是文档或 wiki 相关 scope，那后面命令写得再像样也没用。

### 4. 想一步到位把多维表格也全做完

这个不一定错，但没必要一口吃完。

更稳的路径是：

1. 先把文档链路跑通
2. 再把表格能力接上

因为文档更容易验证，也更适合先当“权限链是否通了”的试金石。

## 如果你现在就想把它写成一条可执行思路

我建议按这个顺序：

1. 先把飞书通信层接通
2. 再确认当前 `lark-cli` 是用户身份
3. 先对一个现有文档做 fetch
4. 再做一次小范围写入验证
5. 文档验证过了，再去扩到多维表格

这个顺序不花哨，但很稳。

## 最后只记住 4 句话

1. 飞书链接不是授权。
2. 聊天能力不是写文档能力。
3. `lark-cli` 不是配角，它就是这条权限链的关键工具。
4. 文档写通以后，多维表格不是重开一局，而是在同一套基础上继续往下接。
